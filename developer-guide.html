This guide gives a developer unfamiliar with kbeatz enough context to make a change, add an endpoint, or extend the frontend without reading the full project CLAUDE.md. It covers the module map, hexagonal architecture in practice, a complete end-to-end feature walkthrough, Gradle composite build structure, domain conventions, frontend development, and CLI commands.

Module Map

kbeatz is a Gradle composite build (includeBuild). Each module is a self-contained Gradle root with its own settings.gradle.kts and build.gradle.kts.

kbeatz-catalog Hexagonal Architecture in Practice

The catalog module follows strict hexagonal architecture (ports and adapters). The dependency direction is inward: adapters depend on the application layer; the application layer depends on domain interfaces; infrastructure implements those interfaces.

Layer rules

Domain layer (domain/)

  • No framework imports (io.ktor., org.jetbrains.exposed., etc.)

  • No java.time.* - use kotlinx.datetime.Instant and kotlinx.datetime.LocalDate

  • No java.util.UUID - use kotlin.uuid.Uuid

  • No null returns from public API methods - use nullable return type (Album?) or throw a domain exception from kbeatz-common

  • Interfaces are the ports: AlbumRepository, TrackRepository

Application layer (application/service/)

  • Contains business logic

  • Depends on domain interfaces, not concrete implementations

  • Throws domain exceptions from kbeatz-common

  • No HTTP-specific types

Adapters (adapters/inbound/web/)

  • Ktor route handlers

  • Calls the OpenAPI-generated Paths stubs and maps to/from domain types

  • Maps ResourceNotFoundException to 404 etc. via StatusPages plugin

Infrastructure (infrastructure/persistence/)

  • Implements repository interfaces using Exposed ORM

  • Maps between Exposed result rows and domain types

  • All database access via suspendTransaction (Exposed coroutine support)

OpenAPI spec is the contract

Before writing any handler, edit the spec:

# Edit the spec first
kbeatz-catalog/api/openapi.yaml

# Then regenerate stubs
./gradlew :kbeatz-catalog:openApiGenerate

The generated stubs appear in kbeatz-catalog/build/generated/api/. Do not edit them manually - they are regenerated on every build.

End-to-End Feature Walkthrough: Adding GET /api/v1/albums/{albumId}/track-listing

This section walks through every step required to add a new endpoint. Follow these steps exactly and a developer unfamiliar with the codebase can add an endpoint without reading CLAUDE.md.

Step 1: Update openapi.yaml

# Open the spec
vim kbeatz-catalog/api/openapi.yaml

Add the path under paths::

/api/v1/albums/{albumId}/track-listing:
  get:
    operationId: getAlbumTrackListing
    summary: Get the track listing for an album
    tags: [Albums]
    parameters:
      - name: albumId
        in: path
        required: true
        schema:
          type: string
          format: uuid
    responses:
      '200':
        description: Track listing
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/TrackListing'
      '404':
        $ref: '#/components/responses/NotFound'

Add the TrackListing schema to components/schemas::

TrackListing:
  type: object
  required: [albumId, tracks]
  properties:
    albumId:
      type: string
      format: uuid
      description: Album identifier
      example: "01933d17-b650-7c4a-a234-c9a91a3afb8c"
    tracks:
      type: array
      description: Ordered list of tracks
      items:
        $ref: '#/components/schemas/TrackSummary'

TrackSummary:
  type: object
  required: [position, title]
  properties:
    position:
      type: string
      description: Track position (e.g. A1, 1, 2)
      example: "A1"
    title:
      type: string
      description: Track title
      example: "Some Song"
    durationSeconds:
      type: integer
      format: int32
      description: Duration in seconds if known
      example: 210

Step 2: Run openApiGenerate

./gradlew :kbeatz-catalog:openApiGenerate

Check kbeatz-catalog/build/generated/api/ for the new generated stubs.

Step 3: Add the domain model

Add TrackListing.kt in domain/model/:

package org.javafreedom.kbeatz.catalog.domain.model

import kotlin.uuid.Uuid

data class TrackListing(
    val albumId: Uuid,
    val tracks: List<TrackSummary>,
)

data class TrackSummary(
    val position: String,
    val title: String,
    val durationSeconds: Int?,
)

Note: no framework imports. Uuid is from kotlin.uuid, not java.util.UUID.

Step 4: Add the repository port

Add the query method to AlbumRepository.kt in domain/repository/:

interface AlbumRepository {
    // ... existing methods ...
    suspend fun findTrackListingByAlbumId(albumId: Uuid): TrackListing?
}

Step 5: Implement the service

Add the method to AlbumService.kt in application/service/:

class AlbumService(private val albumRepository: AlbumRepository) {

    suspend fun getTrackListing(albumId: Uuid): TrackListing =
        albumRepository.findTrackListingByAlbumId(albumId)
            ?: throw ResourceNotFoundException("Album $albumId not found")
}

Import ResourceNotFoundException from kbeatz-common.

Step 6: Add the Exposed ORM query

Add the implementation to ExposedAlbumRepository.kt in infrastructure/persistence/:

override suspend fun findTrackListingByAlbumId(albumId: Uuid): TrackListing? =
    suspendTransaction {
        val album = AlbumsTable
            .selectAll()
            .where { AlbumsTable.id eq albumId }
            .singleOrNull() ?: return@suspendTransaction null

        val tracks = TracksTable
            .selectAll()
            .where { TracksTable.albumId eq albumId }
            .orderBy(TracksTable.position)
            .map { row ->
                TrackSummary(
                    position = row[TracksTable.position],
                    title = row[TracksTable.title],
                    durationSeconds = row[TracksTable.durationSeconds],
                )
            }

        TrackListing(albumId = albumId, tracks = tracks)
    }

Step 7: Add the route handler

Add TrackListingHandler.kt in adapters/inbound/web/albums/:

package org.javafreedom.kbeatz.catalog.adapters.inbound.web.albums

import io.ktor.server.application.call
import io.ktor.server.response.respond
import io.ktor.server.routing.Route
import io.ktor.server.routing.get
import kotlin.uuid.Uuid
import org.javafreedom.kbeatz.catalog.application.service.AlbumService

fun Route.trackListingRoutes(albumService: AlbumService) {
    get("/api/v1/albums/{albumId}/track-listing") {
        val albumId = Uuid.parse(call.parameters["albumId"]!!)
        val listing = albumService.getTrackListing(albumId)
        call.respond(listing.toApiModel())
    }
}

Register the route in plugins/Routing.kt by adding the import and calling the route function inside the route("/api/v1") block:

// plugins/Routing.kt
import org.javafreedom.kbeatz.catalog.adapters.inbound.web.albums.trackListingRoutes

fun Application.configureRouting(/* ... */ albumService: AlbumService /* ... */) {
    routing {
        route("/api/v1") {
            // ... existing routes ...
            trackListingRoutes(albumService)
        }
    }
}

Step 8: Write unit tests

In src/test/kotlin/…​/application/service/:

class AlbumServiceTest {

    private val albumRepository = mockk<AlbumRepository>()
    private val albumService = AlbumService(albumRepository)

    @Test
    fun `should return track listing when album exists`() = runTest {
        val albumId = Uuid.random()
        val listing = TrackListing(albumId, listOf(TrackSummary("A1", "Song", 180)))
        coEvery { albumRepository.findTrackListingByAlbumId(albumId) } returns listing

        val result = albumService.getTrackListing(albumId)

        assertThat(result).isEqualTo(listing)
    }

    @Test
    fun `should throw ResourceNotFoundException when album not found`() = runTest {
        val albumId = Uuid.random()
        coEvery { albumRepository.findTrackListingByAlbumId(albumId) } returns null

        assertThrows<ResourceNotFoundException> {
            albumService.getTrackListing(albumId)
        }
    }
}

Composite Gradle Build Structure

Why includeBuild instead of subprojects

kbeatz uses Gradle’s includeBuild() (composite builds) rather than subprojects. Each module is a separate Gradle root with its own settings.gradle.kts. This means:

  • Each module can be built and tested independently: ./gradlew -p kbeatz-catalog check

  • Each module has its own dependency lock file

  • Modules can be developed and released separately

  • The monorepo root coordinates builds via build.gradle.kts task aggregation

The tradeoff is that each module must carry its own convention plugin configuration. The shared Gradle build logic lives in build-logic/ which is referenced by every module’s settings.gradle.kts:

// kbeatz-catalog/settings.gradle.kts
pluginManagement {
    includeBuild("../build-logic")
    repositories { gradlePluginPortal(); mavenCentral() }
}

Running tasks per module

# Run tests for catalog only
./gradlew :kbeatz-catalog:test

# Run Detekt for all modules
./gradlew :kbeatz-catalog:detektMain :kbeatz-cli:detektMain \
          :kbeatz-common:detektMain :kbeatz-sources:detektMain \
          :kbeatz-tagger:detektMain

# Equivalent shortcut - runs all checks across all modules
./gradlew check

# Build a single module fat JAR
./gradlew :kbeatz-cli:shadowJar

The root ./gradlew commands delegate to all included builds. The :module:task prefix form runs a task in a specific module.

Dependency substitution for kbeatz-common

kbeatz-common is a shared library. When another module declares a dependency on it, the composite build substitutes the Maven artifact reference with the local project:

// root settings.gradle.kts
includeBuild("kbeatz-common") {
    dependencySubstitution {
        substitute(module("org.javafreedom.kbeatz:kbeatz-common"))
            .using(project(":"))
    }
}

This means changes to kbeatz-common are immediately visible to kbeatz-catalog without publishing to a Maven repository first.

Lockfile regeneration

Gradle dependency locking is enabled to make builds reproducible. If you add or update a dependency in a module’s build.gradle.kts, regenerate its lockfiles:

# Regenerate lockfiles for a specific module
./gradlew --project-dir kbeatz-catalog dependencies --write-locks

# Or regenerate for all modules at once
./gradlew --write-locks

Commit the updated gradle.lockfile (and buildscript-gradle.lockfile if present) as part of the same PR that adds the dependency.

Detekt config per module

Each module carries its own config/detekt/detekt.yml. This is required because each includeBuild is a separate Gradle root - project.rootDir inside the convention plugin resolves to the module directory, not the monorepo root. A missing detekt config causes:

Provided path '<module>/config/detekt/detekt.yml' does not exist!

When scaffolding a new module, copy the config from an existing module:

mkdir -p new-module/config/detekt
cp kbeatz-catalog/config/detekt/detekt.yml new-module/config/detekt/detekt.yml

Domain Conventions

Date and time types

Use kotlinx.datetime throughout domain code:

Use this Never this

kotlinx.datetime.Instant

java.time.Instant

kotlinx.datetime.LocalDate

java.time.LocalDate

kotlin.uuid.Uuid

java.util.UUID

java.time.* and java.util.UUID may appear in infrastructure and persistence layers where Exposed ORM requires them for type conversion, but never in domain or application layers.

Domain exceptions

All domain exceptions are defined in kbeatz-common/DomainExceptions.kt. Throw them from the service layer; never from route handlers or repositories:

// Service layer - throw domain exception
suspend fun getAlbum(id: Uuid): Album =
    albumRepository.findById(id) ?: throw ResourceNotFoundException("Album $id not found")

// Never throw from route handler - let StatusPages convert exceptions
get("/api/v1/albums/{id}") {
    val album = albumService.getAlbum(Uuid.parse(call.parameters["id"]!!))
    call.respond(album.toResponse())
}

The StatusPages plugin in plugins/StatusPages.kt maps domain exceptions to HTTP codes:

Exception HTTP Status

ResourceNotFoundException

404

BusinessValidationException

422

ConflictException

409

AuthenticationException

401

AuthorizationException

403

Null safety in public APIs

Public method return types must be honest. If a method can fail to find a result, return a nullable type rather than throwing:

// Repository interface - returns null when not found
interface AlbumRepository {
    suspend fun findById(id: Uuid): Album?       // nullable - caller decides
    suspend fun save(album: Album): Album        // never null
}

// Service layer - converts null to exception with context
suspend fun getAlbum(id: Uuid): Album =
    albumRepository.findById(id)
        ?: throw ResourceNotFoundException("Album $id not found")

Frontend: Adding a UI Feature

Setup

cd kbeatz-ui
npm install

Regenerating the TypeScript client

The TypeScript client in src/api/generated/ is auto-generated from kbeatz-catalog/api/openapi.yaml. Never edit it manually. After any API change:

npm run api:generate

The generated client uses Axios. Import from the generated barrel:

import { AlbumsApi, Configuration } from '../api/generated';

const api = new AlbumsApi(new Configuration({ basePath: '/api/v1' }));
const albums = await api.listAlbums({ page: 0, size: 20 });

State management

kbeatz-ui uses TanStack Query for all server state and React useState / useReducer for local ephemeral UI state. There is no Redux or global state library. Server-state fetching via useEffect is forbidden; use useQuery from TanStack Query instead (see ADR-010).

// Good - TanStack Query manages the fetch, cache, and loading state
function useAlbum(albumId: string) {
    return useQuery({
        queryKey: ['album', albumId],
        queryFn: () => api.getAlbum({ albumId }),
    });
}

function AlbumDetail({ albumId }: { albumId: string }) {
    const { data: album, isPending, isError } = useAlbum(albumId);

    if (isPending) return <Loading />;
    if (isError || !album) return <NotFound />;
    return <AlbumView album={album} />;
}

Adding a new feature

# Create the feature directory
mkdir src/features/my-feature

# Recommended structure
src/features/my-feature/
├── MyFeature.tsx          # main component
├── MyFeature.test.tsx     # Vitest + React Testing Library tests
└── index.ts               # barrel export

Register the route in App.tsx if the feature needs its own URL.

Testing

# Run all tests
npm run test

# Run with coverage
npm run test:coverage

# Run in watch mode
npm run test -- --watch

Tests use Vitest and React Testing Library. Mock API calls with vi.mock or MSW.

Build

npm run build

TypeScript strict mode is enforced (noUnusedLocals: true, noUnusedParameters: true, strictNullChecks: true). The build fails on any type error. Fix all errors before opening a PR.

Internationalisation (i18n) Conventions for kbeatz-ui

All user-visible strings in kbeatz-ui are managed through react-i18next. This section explains how to add translated strings, follow key naming conventions, format dates, and verify that no bare strings slip into the codebase.

Overview

kbeatz-ui uses a single translation namespace backed by one source file:

kbeatz-ui/src/locales/en.json      <-- add all string keys here
kbeatz-ui/src/lib/i18n.ts          <-- i18next setup + formatDate / formatDateTime helpers

The t() function from useTranslation() resolves keys at runtime. The eslint-plugin-i18next ESLint plugin enforces that no bare string literals appear in JSX.

How to add a new translated string

Step 1 - Before (bare string in JSX, will fail lint):

function ScanTrigger() {
  return <button type="button">Scan library</button>
}

Step 2 - Add the key to src/locales/en.json:

{
  "library": {
    "scanTrigger": {
      "buttonLabel": "Scan library"
    }
  }
}

Step 3 - After (use t() in the component):

import { useTranslation } from 'react-i18next'

function ScanTrigger() {
  const { t } = useTranslation()
  return <button type="button">{t('library.scanTrigger.buttonLabel')}</button>
}

That is the complete workflow. No configuration changes are needed.

Namespace structure and key naming

kbeatz-ui uses a single namespace (translation). All keys live in en.json under a feature-based hierarchy:

<feature>.<component>.<label>

Examples from the existing codebase:

Key Value

albumCard.unknownArtist

"Unknown Artist"

albumGrid.noResults

"No albums found…​"

scanProgress.running

"Scanning: {{progress}} albums"

editableField.saveFailed

"Save failed"

Rules: - Use camelCase for each segment. - Group by feature first (albums, library, sync, common). - Keep keys short but descriptive - buttonLabel not btn. - For interpolated values use double-brace syntax: "Hello, {{name}}".

How to run the missing-key test

A Vitest test in src/lib/i18n.test.ts walks all leaf keys in en.json and verifies each one resolves to a non-empty string. Run it with:

cd kbeatz-ui

# Run all tests (includes the missing-key test)
npm run test

# Run only the i18n test file by filename filter
npm run test -- src/lib/i18n.test.ts

# Or run with verbose output to see each key checked
npm run test -- --reporter=verbose

If you add a key to en.json but misspell it in the component (or vice versa), the test will fail with a list of missing or unresolved keys.

Date and timestamp formatting

The API returns dates in three formats. Do not format them inline with Intl.DateTimeFormat in components. Use the helpers from src/lib/i18n.ts:

Input shape Example Use

Bare year

"1978"

formatDate("1978") returns "1978" unchanged

Year-month

"1978-06"

formatDate("1978-06") returns locale month+year, e.g. "June 1978"

Full ISO date

"1978-07-14"

formatDate("1978-07-14") returns locale short date, e.g. "14 Jul 1978"

UTC timestamp

"2026-06-09T20:31:20Z"

formatDateTime("2026-06-09T20:31:20Z") returns locale date+time

Non-ISO strings (e.g. "circa 1970") are returned unchanged without throwing.

Correct usage in a component:

import { formatDate } from '../../lib/i18n'

function AlbumMeta({ date }: { date?: string }) {
  return date ? <span>{formatDate(date)}</span> : null
}

Incorrect usage (bypasses locale, fails review):

// Do not do this
<span>{new Intl.DateTimeFormat().format(new Date(date))}</span>

When a field is editable (e.g. the DATE field in album-detail), pass displayValue to EditableField so the formatted date is shown while the raw tag value is preserved in the edit input:

<EditableField
  value={album.date}
  displayValue={album.date !== undefined ? formatDate(album.date) : undefined}
  fieldName="DATE"
  onSave={handleAlbumTagSave}
/>

How to verify no bare strings

npm run lint runs ESLint with eslint-plugin-i18next. Any string literal directly inside JSX triggers the i18next/no-literal-string rule and fails the lint step:

cd kbeatz-ui
npm run lint

The only allowed exceptions are non-translatable content such as icon characters or aria-label decorations, which must be suppressed with an explicit comment:

{/* eslint-disable-next-line i18next/no-literal-string */}
<text x="100" y="90" textAnchor="middle" fill="#666" fontSize="40">music</text>

SVG <text> elements used as decorative icons are a common case where suppression is appropriate. The rule also fires on SVG content because JSX does not distinguish between HTML and SVG elements. Always add the eslint-disable-next-line comment immediately before the SVG element, not at the top of the file.

If the lint output shows a no-literal-string error on a line you did not expect, check whether the string is truly non-translatable (icon glyph, number, punctuation). If it carries meaning for the user, move it to en.json.

Checklist before committing i18n changes

  • New key added to src/locales/en.json under the correct feature group

  • Key uses feature.component.label naming convention

  • Component uses const { t } = useTranslation() and calls t('key')

  • Dates use formatDate() or formatDateTime() from src/lib/i18n.ts

  • npm run lint passes (no bare-string warnings)

  • npm run test — src/lib/i18n.test.ts passes (no missing keys)

  • npm run build passes (TypeScript strict mode)

CLI: Adding a Command

The CLI module (kbeatz-cli) uses Clikt for command-line argument parsing. It is distributed as a fat JAR built by the Gradle Shadow plugin.

Clikt entry point structure

// KbeatzTaggerCli.kt - the root Clikt command
class KbeatzTaggerCli : CliktCommand(name = "kbeatz") {
    override fun run() = Unit
}

fun main(args: Array<String>) = KbeatzTaggerCli()
    .subcommands(
        TagAlbumsCommand(/* dependencies */),
        MigrateIdFilesCommand(/* dependencies */),
    )
    .main(args)

Adding a subcommand

  1. Create a new file in command/:

    package org.javafreedom.kbeatz.cli.command

import com.github.ajalt.clikt.core.CliktCommand
import com.github.ajalt.clikt.parameters.options.option
import com.github.ajalt.clikt.parameters.options.required
import com.github.ajalt.clikt.parameters.types.path

class ExportCommand : CliktCommand(
    name = "export",
    help = "Export tag data for an album to a YAML file",
) {
    private val albumDir by option("--album-dir", help = "Album directory path")
        .path(mustExist = true, canBeFile = false)
        .required()

    override fun run() {
        // implementation
    }
}

  2. Register it in the root command’s subcommands() call in KbeatzTaggerCli.kt.

Building the fat JAR

./gradlew :kbeatz-cli:shadowJar

The output JAR is at kbeatz-cli/build/libs/kbeatz-cli-<version>-all.jar. Run it with:

java -jar kbeatz-cli/build/libs/kbeatz-cli-*-all.jar --help

GitHub Pages: Clean Deploy and Stale File Cleanup

How the clean deploy works

The AsciiDoc documentation is published to GitHub Pages by the release.yml workflow using force_orphan: true. This setting recreates the gh-pages branch from scratch on every release so that renamed or deleted .adoc source files do not leave stale HTML pages accessible indefinitely.

Quality reports (Detekt, ESLint, Trivy) are published by publish-docs.yml on every push to main. That workflow uses keep_files: true so that the quality reports are overlaid on top of the existing docs without wiping them. Because report files use fixed names (detekt/${module}.html, eslint/eslint-report.html), no stale report files accumulate.

What happens when a page is renamed

  1. Rename docs/old-name.adoc to docs/new-name.adoc and merge to main.

  2. When the next release is cut, release.yml runs asciidoctor and deploys with force_orphan: true.

  3. The gh-pages branch is recreated with only the current HTML output. old-name.html no longer exists; new-name.html is present.

  4. publish-docs.yml runs on the same push and overlays the quality reports back onto the branch.

Manual fallback: resetting gh-pages

If the gh-pages branch ever accumulates unwanted files outside of a release cycle, reset it manually:

# Delete the remote gh-pages branch
git push origin --delete gh-pages

# Then trigger a re-publish in one of two ways:
# Option A - re-run the release workflow from the GitHub Actions UI
# Option B - push a commit to main (publish-docs.yml runs on every push and will
#             recreate the branch with quality reports; docs are restored on the
#             next release)
