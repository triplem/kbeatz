= ADR-CAT-001 - AlbumDetail Typed Fields for Country and Media Format
:revnumber: 1.0
:revdate: 2026-06-18
:author: kbeatz team
:status: Accepted

== Status

Accepted

== Context

The Discogs release page exposes two metadata fields that collectors find
immediately useful when identifying a pressing: the country of manufacture
(e.g., "USA & Canada", "Europe") and the media format description
(e.g., "2 x Vinyl, LP, Album, Red Translucent"). Users want to see these
at a glance on the album detail page without leaving the app.

The Discogs sync feature (kbeatz-sources) already receives this data.
The question is how to expose it through the AlbumDetail API schema:
as explicit typed fields, or as a generic extra-tags map.

Vorbis Comment tags carry this data on disk:

* `COUNTRY` - country/region of manufacture
* `MEDIA` - physical media format

== Decision

Add `country` and `mediaFormat` as explicit optional String fields directly
on the `Album` domain model and the `AlbumDetail` / `Album` schemas in
the OpenAPI specification.

These fields follow the same pattern as existing fields such as `genre`,
`label`, and `catalogNumber`.

== Rationale

Typed fields were chosen over a generic map for the following reasons:

* **Type safety**: The compiler and OpenAPI tooling enforce the field
  contract. A `Map<String, String>` gives no compile-time guarantee that
  `country` is present or correctly named.
* **OpenAPI discoverability**: Frontend developers and API consumers can
  see `country` and `mediaFormat` directly in the spec without inspecting
  tag documentation.
* **Frontend filtering**: Typed fields allow the UI to filter or sort by
  country without parsing a free-form map or knowing which key to look for.
* **Consistency**: All other business-significant Vorbis Comment fields
  (`title`, `artist`, `albumArtist`, `genre`, `label`, `catalogNumber`)
  are already typed fields on the domain model.

== Consequences

* Each additional Vorbis Comment field to be surfaced in the API requires
  a domain model change, an OpenAPI schema update, a Liquibase migration,
  and a mapper update. This is the accepted trade-off for v1 scope.
* The OpenAPI specification remains the single source of truth for the
  fields available to frontend consumers.
* Fields that are absent from the FLAC tags (or not yet populated by the
  Discogs sync) are represented as `null` / omitted in the API response,
  and the frontend renders them conditionally.

== Alternatives Considered

[cols="1,2,2", options="header"]
|===
|Option |Pros |Cons

|Typed fields on AlbumDetail (this ADR)
|Type-safe; discoverable in OpenAPI spec; consistent with existing fields;
 enables frontend filtering without key knowledge
|Each new field requires a coordinated change across domain, spec, schema,
 and mapper layers

|Generic extra-tags map (`Map<String, String>`)
|Accept any Vorbis Comment key-value pair with no schema change per field;
 flexible for future fields
|Loses type safety; makes the OpenAPI spec opaque to consumers; requires
 frontend to know which map keys exist and validate them at runtime;
 makes filtering and sorting awkward; diverges from existing field pattern

|Separate raw-tags endpoint (`GET /albums/{id}/tags`)
|Surfaces all Vorbis Comment tags without polluting the main schema;
 useful for a future full tag inspection use case
|Over-engineering for v1 scope; the two fields being added have immediate
 UI value and are well-defined; a raw-tags endpoint can be considered in
 a future epic if a broader tag inspection use case arises

|===
