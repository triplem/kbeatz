== Overview

This guide covers day-to-day operations for `kbeatz-catalog`: health probe configuration
for Kubernetes, configuring the Discogs integration, monitoring the image quota, and
troubleshooting common errors.

== Health Probes

kbeatz-catalog exposes three health endpoints at the HTTP root (not under `/api/v1`).
K8s-idiomatic path names (`/livez`, `/readyz`) are used per the implementation decision
for issue #293, rather than the `/health/live` and `/health/ready` paths originally
described in the acceptance criteria.

[cols="2,1,3", options="header"]
|===
| Endpoint | Purpose | Use as

| `GET /livez`
| Liveness
| `livenessProbe` - always 200 while the process is running; never checks the DB.

| `GET /readyz`
| Readiness
| `readinessProbe` - 200 when DB is reachable; 503 with `ErrorResponse` body when not.

| `GET /healthz`
| Combined (legacy)
| Not recommended for new deployments; retained for backwards compatibility.
|===

All probe endpoints return JSON and do not require authentication.

=== Response Shapes

*Liveness (200)*

[source,json]
----
{"status":"UP"}
----

*Readiness (200)*

[source,json]
----
{"status":"UP"}
----

*Readiness (503) - DB unreachable*

[source,json]
----
{"code":"DATABASE_UNAVAILABLE","message":"Database is not reachable"}
----

=== Kubernetes Probe Configuration

The example below configures all three probe types. The `startupProbe` handles JVM cold-start:
it gives the process up to 5 minutes (30 checks x 10 s interval) to become live before the
`livenessProbe` takes over.

[source,yaml]
----
containers:
  - name: kbeatz-catalog
    image: kbeatz-catalog:latest
    ports:
      - containerPort: 8080

    startupProbe:
      # Allow up to 5 minutes for JVM cold-start before liveness kicks in.
      httpGet:
        path: /livez
        port: 8080
      initialDelaySeconds: 10
      periodSeconds: 10
      failureThreshold: 30
      timeoutSeconds: 2

    livenessProbe:
      # Restart the pod only when the process is truly unresponsive.
      httpGet:
        path: /livez
        port: 8080
      periodSeconds: 15
      failureThreshold: 3
      timeoutSeconds: 2

    readinessProbe:
      # Remove from load balancer when the DB is unreachable.
      httpGet:
        path: /readyz
        port: 8080
      periodSeconds: 10
      failureThreshold: 2
      timeoutSeconds: 2
----

NOTE: The `startupProbe` and `livenessProbe` both use `/livez` intentionally.
The liveness probe must not check the database - if the DB is down the pod should remain
running (so it can reconnect) rather than be restarted.

== kbeatz-ui Configuration

=== LAN Deployment - API Base URL

By default the UI sends all API requests to a relative path (`/api/v1`), which works
when the UI and the catalog backend are served through the same origin (e.g. an nginx
reverse proxy on the same machine).

When the UI is served from a *different* host or port than the catalog backend (for
example, the UI is hosted on a NAS but the catalog runs on a separate server), set the
`VITE_API_BASE_URL` environment variable at *build time*:

[source,bash]
----
# Build the UI pointing at a specific LAN host
VITE_API_BASE_URL=http://192.168.1.10:8080/api/v1 npm run build
----

Or via a `.env.local` file in `kbeatz-ui/` (gitignored):

[source,bash]
----
# kbeatz-ui/.env.local
VITE_API_BASE_URL=http://192.168.1.10:8080/api/v1
----

NOTE: `VITE_API_BASE_URL` is a Vite build time variable. It is embedded into the
compiled JavaScript bundle. If you change the catalog server address, you must rebuild
the UI and redeploy. For most home/LAN deployments the default reverse-proxy approach
(UI and catalog served from the same origin) is simpler and requires no variable.

[cols="2,1,3", options="header"]
|===
| Variable | Required | Description

| `VITE_API_BASE_URL`
| No (default: `/api/v1`)
| Absolute URL prefix for all catalog API calls. Set only when the UI and catalog are
served from different origins. Example: `http://192.168.1.10:8080/api/v1`.
|===

== Environment Variables

[cols="2,1,3", options="header"]
|===
| Variable | Required | Description

| `CATALOG_LIBRARY_ROOT`
| Yes
| Absolute path to the root of the FLAC music library. The service refuses to start if this is not set.

| `DISCOGS_TOKEN`
| No (but recommended)
| Discogs Personal Access Token (PAT) for metadata sync. Without this, all Discogs sync operations will fail with a 503-style error. See <<discogs-token>>.

| `CATALOG_JDBC_URL`
| No
| JDBC URL for the H2 database. Defaults to `jdbc:h2:file:./data/kbeatz;DB_CLOSE_DELAY=-1;MODE=PostgreSQL`. Override for custom data directory or PostgreSQL migration target.

| `CATALOG_DATA_DIR`
| No (default: `./data`)
| Directory for runtime data files including `discogs-image-quota.json`. Created automatically on startup if it does not exist. Must be writable by the service process.

| `CATALOG_REPAIR_TIMEOUT_SECONDS`
| No (default: `60`)
| Maximum wall-clock seconds the startup lock-file repair phase may run before it is aborted and startup continues. Any lock files not repaired within this window are left in place and will be picked up automatically on the next scan request. See <<startup-repair-failed>>.

| `CATALOG_SCAN_PARALLELISM`
| No (default: `4`)
| Number of parallel threads used during a full library scan. Increase on hosts with fast I/O and many CPU cores; decrease on resource-constrained machines.
|===

WARNING: Never log, print, or include `DISCOGS_TOKEN` in error messages or API responses.
The token grants access to your Discogs account and must be treated as a secret.

[#discogs-token]
== Configuring the Discogs Token

=== Obtaining a Personal Access Token

. Log in to https://www.discogs.com/
. Navigate to **Settings → Developers → My Applications → Generate new token**
. Copy the generated token - you will not be able to view it again

=== Setting the Token

==== Docker

[source,bash]
----
docker run -e DISCOGS_TOKEN=your_token_here \
           -e CATALOG_LIBRARY_ROOT=/music \
           -v /mnt/music:/music \
           kbeatz-catalog:latest
----

==== Docker Compose

[source,yaml]
----
services:
  catalog:
    image: kbeatz-catalog:latest
    environment:
      DISCOGS_TOKEN: ${DISCOGS_TOKEN}      # from .env file
      CATALOG_LIBRARY_ROOT: /music
    volumes:
      - /mnt/music:/music
----

Store the token in a `.env` file (gitignored) and reference it with `${DISCOGS_TOKEN}`.

==== Systemd

[source,ini]
----
[Service]
Environment="DISCOGS_TOKEN=your_token_here"
Environment="CATALOG_LIBRARY_ROOT=/mnt/music"
----

=== Verifying the Token

A correctly configured service logs this at startup:

[source,text]
----
INFO  o.j.k.catalog.AppConfig - Discogs integration enabled
----

A missing token produces:

[source,text]
----
WARN  o.j.k.catalog.AppConfig - DISCOGS_TOKEN not set - Discogs sync will be unavailable
----

== Discogs Image Quota Monitoring

The Discogs API permits **1 000 image downloads per day** per PAT. kbeatz-catalog
tracks this quota in a JSON file to preserve the count across service restarts.

=== Quota File Location

`$DATA_DIR/discogs-image-quota.json`

Default location (`CATALOG_DATA_DIR` defaults to `./data`):

[source,text]
----
./data/discogs-image-quota.json
----

NOTE: The `CATALOG_DATA_DIR` directory is created automatically at startup if it does not exist.
The quota file itself is created on the first image download.

=== Quota File Format

[source,json]
----
{"downloaded":42,"date":"2026-06-07"}
----

[cols="1,3", options="header"]
|===
| Field | Description

| `downloaded` | Number of images downloaded today (UTC)
| `date` | Today's UTC date (ISO 8601). If this differs from today, the counter is reset to 0 on next access.
|===

[#when-the-quota-resets]
=== When the Quota Resets

The quota resets automatically at **UTC midnight**. The reset is triggered lazily:
it happens on the next `canDownload()` or `recordDownload()` call after midnight.

To find when the next reset occurs:

[source,bash]
----
date -u -d "tomorrow 00:00:00" +%Y-%m-%dT%H:%M:%SZ
----

=== Checking Current Usage

[source,bash]
----
cat $DATA_DIR/discogs-image-quota.json
# {"downloaded":750,"date":"2026-06-07"}
----

=== Monitoring via Logs

The service logs quota exhaustion at WARN level:

[source,text]
----
WARN  D.DiscogsMetadataSource - Discogs daily image quota exhausted - skipping image download for release 12345
----

And daily resets at INFO level:

[source,text]
----
INFO  D.DiscogsImageQuota - Discogs image quota reset for new UTC day: 2026-06-08 (was 2026-06-07)
----

=== Manual Quota Reset

To reset the quota before UTC midnight (e.g. after rotating to a new PAT):

[source,bash]
----
# Stop the service first to avoid concurrent writes
systemctl stop kbeatz-catalog

echo '{"downloaded":0,"date":"'$(date -u +%Y-%m-%d)'"}' > $DATA_DIR/discogs-image-quota.json

systemctl start kbeatz-catalog
----

== Rate Limit Behaviour

The Discogs API limits **authenticated metadata requests to 60/minute**.
kbeatz-catalog uses a token-bucket rate limiter to enforce this transparently.

* Requests that exceed the 60/min limit are **suspended** (queued), never rejected.
* At peak throughput, the user may notice a sync operation taking longer than expected.
  This is normal - the limiter is protecting the Discogs API limit.
* The rate-limit state is in-memory and resets when the service restarts. There is no
  persistent token-bucket state.

== Troubleshooting

=== Sync Returns "No Discogs token configured"

*Cause*: `DISCOGS_TOKEN` environment variable is not set.

*Fix*: Set the variable and restart the service. See <<discogs-token>>.

=== Sync Returns "Image quota exhausted"

*Cause*: 1 000 images have been downloaded today.

*Fix*: Wait for UTC midnight (see <<when-the-quota-resets>>). For urgent cases,
manually reset the quota file as described in the manual reset section.

=== "Failed to persist Discogs image quota" in Logs

*Cause*: The service cannot write to `$DATA_DIR/discogs-image-quota.json`. Common
causes: wrong file permissions or a read-only filesystem. The directory itself is
created automatically at startup.

*Fix*:

[source,bash]
----
chmod 755 $DATA_DIR
# Restart the service - it will create the file on first write
----

=== Rate Limit Errors (HTTP 429) from Discogs

*Cause*: Multiple service instances are running simultaneously and sharing the same PAT,
or the token-bucket state was reset (service restart) during a burst.

*Fix*: Ensure only one instance of kbeatz-catalog runs per PAT. If running multiple
instances is required, each needs its own Discogs PAT.

[#startup-repair-failed]
=== "Startup repair failed" / "Startup repair timed out" in Logs

*Cause*: One or more `.kbeatz-write.lock` files were found in the library at startup.
Two distinct situations produce log entries:

* *Per-directory failure* (`"Startup repair failed for <dir>"`) - re-indexing a specific
  album directory failed, typically because the directory was moved or FLAC files were
  deleted while the lock file remained.
* *Timeout* (`"Startup repair timed out after Ns"`) - the repair phase did not finish
  within the configured timeout (default 60 seconds; controlled by
  `CATALOG_REPAIR_TIMEOUT_SECONDS`). Startup continues; unrepaired directories are
  listed in the ERROR log entry.

*Self-healing behaviour*: In both cases the service starts normally. Lock files that
were not repaired at startup are picked up automatically on the next
`POST /api/v1/library/scan` request (and again on the next startup). No operator
action is required unless the lock is stale or the album directory no longer exists.

*Manual cleanup - only when needed*: If a lock file refers to a directory that has
been permanently removed, or if you need to clear a lock immediately rather than
waiting for the next scan, delete the file manually:

[source,bash]
----
# Identify unrepaired lock files
find $CATALOG_LIBRARY_ROOT -name ".kbeatz-write.lock"

# Remove a specific lock file after confirming the directory is gone or corrupt
rm /path/to/album/.kbeatz-write.lock

# No restart needed - the next scan will process remaining directories normally
----

To extend the repair window and reduce timeout occurrences on large libraries, increase
`CATALOG_REPAIR_TIMEOUT_SECONDS` (e.g. `CATALOG_REPAIR_TIMEOUT_SECONDS=300` for a
five-minute window).

== Backup and Restore

=== What to Back Up

[cols="2,3", options="header"]
|===
| Path | Contents

| `./data/kbeatz.mv.db` (default) | H2 database - album index, track metadata, all tag data
| `$DATA_DIR/discogs-image-quota.json` | Daily image quota counter (low value, recreated automatically)
|===

=== Restore Procedure

. Stop the service.
. Replace `kbeatz.mv.db` with the backup copy.
. Start the service. Liquibase migrations run automatically on startup.
. The quota file is optional - the service starts fresh if it is absent.

== Request Tracing (X-Trace-Id)

Every HTTP response includes an `X-Trace-Id` header that can be used to correlate client
errors with server log entries.

=== Behaviour

* If the request includes an `X-Trace-Id` header, the same value is echoed back in the
  response. The caller can supply its own correlation ID (e.g. a frontend request ID or an
  end-to-end trace ID from another system).
* If no `X-Trace-Id` header is present, the service generates a random UUID, uses it
  internally for the request lifetime, and echoes it in the response.

The traceId appears in every structured log line as the `traceId` field:

[source,json]
----
{"timestamp":"2026-06-07T14:32:01Z","level":"DEBUG","traceId":"a1b2c3...","message":"Resource not found"}
----

=== Finding the Server Log for a Client Error

If a client reports an error, ask them for the `X-Trace-Id` from the failing response and
search the logs:

[source,bash]
----
# Grep log output directly
grep "a1b2c3..." /var/log/kbeatz-catalog.log

# If using a log aggregation system (e.g. Loki, Elasticsearch):
# filter by field: traceId = "a1b2c3..."
----

=== Trust Model and v2 Consideration

In v1, kbeatz-catalog is deployed on a trusted LAN without authentication. Echoing
server-generated trace UUIDs to all clients is acceptable in this context.

NOTE: Before any internet-facing deployment (Keycloak JWT/OIDC is planned for Phase 3+, not v1 or v2),
the X-Trace-Id echo behaviour should be re-evaluated. Returning server-generated UUIDs to
unauthenticated clients could allow UUID sequencing analysis. Options include: echoing
only client-supplied IDs (not generated ones), or restricting the header to authenticated
requests only.

== Filter Query Performance

=== LOWER() predicates and index usage

`ExposedAlbumRepository.buildWhereClause()` wraps filter columns in `LOWER()` before
applying `LIKE` predicates (for case-insensitive browsing). Standard B-tree indexes on
the raw column values cannot serve these queries because the function call defeats index
use.

==== H2 v1 (current)

H2 does not support functional (expression-based) indexes. On H2, filtered list queries
scan all rows and apply the `LOWER(LIKE)` predicate in-memory. Performance degrades
linearly with collection size. On a typical home collection (under 20,000 albums) this
is not noticeable, but it becomes measurable at larger scales.

*There is no workaround for H2.* The migration to PostgreSQL (v2) is the correct fix.

==== PostgreSQL v2 (migration target)

When migrating to PostgreSQL, Liquibase changeset `V1-009-albums-lower-functional-indexes`
automatically creates three functional indexes:

[source,sql]
----
CREATE INDEX idx_albums_lower_album_artist ON albums (LOWER(album_artist));
CREATE INDEX idx_albums_lower_composer     ON albums (LOWER(composer));
CREATE INDEX idx_albums_lower_genre        ON albums (LOWER(genre));
----

These indexes allow PostgreSQL to satisfy `WHERE LOWER(album_artist) LIKE 'foo%'` via
an index scan rather than a sequential scan.

NOTE: The changeset uses a Liquibase `preConditions` block with `onFail: MARK_RAN` and
`dbms: postgresql` so H2 silently skips it. No manual intervention is required when
running on H2; the indexes are applied automatically when the database is PostgreSQL.

==== Free-text search ('q' parameter)

The `q` free-text filter ORs four leading-wildcard `LOWER(LIKE)` predicates across
`album_artist`, `album`, `composer`, and `label`. Neither B-tree indexes nor functional
indexes can serve leading-wildcard LIKE patterns efficiently. On PostgreSQL, a GIN index
on a `tsvector` column would be needed, but that also requires changes to the query
layer. This is tracked as future work and is not addressed in v2.

== Upgrade Path

kbeatz-catalog uses Liquibase for schema migrations. On every startup:

. Liquibase checks the `DATABASECHANGELOG` table.
. Any pending migrations are applied in version order.
. The service starts accepting requests after migrations complete.

No manual SQL is required for upgrades. Always back up the database before deploying
a new version to a production instance.
