= User Guide: Tag Editing in the Album Detail View
:toc:
:toc-placement: preamble
:icons: font

This guide explains how to navigate to an album's detail view, edit individual tag fields inline, and understand the atomic write guarantees that protect your FLAC files from corruption.

== Overview

kbeatz provides in-place tag editing directly in the browser.
Any Vorbis Comment field can be updated without leaving the album detail view.
Changes are written atomically to the FLAC files on disk. The original file is never left in a partially-written state.

The following fields are editable:

.Album-level fields (written to all FLAC files in the album directory)
[cols="1,3",options="header"]
|===
| Field | Vorbis Comment tag
| Album | `ALBUM`
| Album Artist | `ALBUMARTIST`
| Date | `DATE`
| Genre | `GENRE`
| Label | `LABEL`
| Catalog # | `CATALOGNUMBER`
| Composer | `COMPOSER`
| Conductor | `CONDUCTOR`
| Ensemble | `ENSEMBLE`
|===

.Track-level fields (written to a single FLAC file)
[cols="1,3",options="header"]
|===
| Field | Vorbis Comment tag
| Title | `TITLE`
| Track # | `TRACKNUMBER`
| Artist | `ARTIST`
|===

== Navigating to Album Detail

. Open kbeatz in a browser (default: `http://localhost:3005`). The *Albums* page is the default
  destination; you can also reach it from the navigation drawer in the App Bar.
. Browse or search for the album you want to edit in the paginated album grid.
. Click on any album card to navigate to the detail view at `/albums/{albumId}`.
+
The detail view shows all tag fields for the album and a table of its tracks. The detail view,
like the rest of the UI, is built on Material Design (MUI) and is fully responsive: on a phone
the tag fields and track table stack into a single column.

When you return to the grid from the detail view, your previous page, scroll position, and
active filters are restored.

== Editing Album-Level Tags

Album-level tags are written to *all FLAC files* in the album directory simultaneously.
Use these fields to update metadata that applies to the release as a whole (e.g. genre, label, date).

=== Batch-Save Flow

kbeatz uses a *batch-save* flow for tag edits.
Changes you make are held in a "pending" (dirty) state in the browser until you explicitly commit them all at once with the Save button.

* *Enter or Tab* - stages the typed value as a pending change.
  The field shows a pending indicator so you can see which fields have unsaved changes.
  No API call is made at this point.
* *Blur (click outside the input)* - *discards* the pending change for that field.
  A brief "Changes discarded - press Enter to save" hint appears.
* *Save button* - commits all pending changes.
  The button shows a badge with the number of dirty fields.
  A confirmation dialog appears before any file write is performed.

=== Procedure

. In the *Album Tags* section of the detail view, locate the field you want to change (e.g. `Genre`).
. Click on the displayed value.
  An inline text input appears, pre-filled with the current value.
. Type the new value.
. Press *Enter* or *Tab* to stage the change as pending.
  The field shows a pending indicator. No API call is made yet.
+
NOTE: Clicking outside the input (blur) *discards* the change for that field.
A "Changes discarded - press Enter to save" hint appears briefly.
+
. Repeat steps 2-4 for any other album-level fields you want to change.
. Click the *Save* button (the badge shows the number of pending fields).
  A confirmation dialog appears listing the fields that will be written.
. Confirm the dialog to commit all pending changes.
  A `PATCH /api/v1/albums/{albumId}` request is sent and the tags are written to every FLAC file in the album directory.
  The pending indicators are cleared and the fields display the saved values.
. To discard *all* pending changes without saving, navigate away from the page.
  No API call is made.

=== Error Handling

If the save fails (for example, the backend cannot write to disk), the pending changes remain staged and an error message appears above the Save button.
Correct the problem (for example, check disk permissions or restart the catalog service) and click Save again to retry.

== Editing Track-Level Tags

Track-level tags are written to a *single FLAC file* (the file corresponding to the specific track).
Use track-level editing to fix per-track fields such as title or track artist in VA (Various Artists) compilations.

=== Procedure

Track-level edits follow the same batch-save flow as album-level edits.
Changes are staged as pending and committed together with the Save button.

. In the *Tracks* table at the bottom of the detail view, locate the track and field you want to change (Title, Track #, or Artist).
. Click on the displayed value in that cell.
  An inline text input appears, pre-filled with the current value.
. Type the new value.
. Press *Enter* or *Tab* to stage the change as pending.
  The cell shows a pending indicator. No API call is made yet.
+
NOTE: Clicking outside the cell (blur) *discards* the change for that cell.
A "Changes discarded - press Enter to save" hint appears briefly.
+
. Repeat steps 2-4 for any other track fields or tracks you want to change.
. Click the *Save* button (the badge shows the total number of pending changes across all fields and tracks).
  A confirmation dialog appears before any file write is performed.
. Confirm the dialog to commit the changes.
  A `PATCH /api/v1/albums/{albumId}/tracks/{trackId}` request is sent for each dirty track field,
  and the tag is written to that track's FLAC file only.
. Navigate away from the page to discard all pending changes without saving.

=== VA Albums (Various Artists)

On VA compilations, the `ARTIST` field is meaningful per track.
Click the Artist cell on any track row to edit that track's `ARTIST` tag without affecting the album-level `ALBUMARTIST`.

== Atomic Write Safety

kbeatz guarantees that *no FLAC file is ever left in a corrupt or partially-written state*.

=== How Writes Work

For each FLAC file that needs to be updated:

. The updated file is written to a temporary file in the same directory (`.tmp` suffix).
. The temporary file is flushed to disk (`fsync`).
. The temporary file is atomically renamed over the original using the operating system's rename system call, which is atomic on all supported Linux filesystems.

If the process is interrupted between steps 1 and 3, the original FLAC file is untouched.
The temporary `.tmp` file is left behind and can be safely deleted.

=== Album-Level Writes and the Write-Lock Manifest

When writing album-level tags across multiple FLAC files, kbeatz creates a `.kbeatz-write.lock` file in the album directory before the first file write.
The manifest contains the list of target file paths.

[source]
----
/music/Jazz/Miles Davis/Kind of Blue/01 So What.flac
/music/Jazz/Miles Davis/Kind of Blue/02 Freddie Freeloader.flac
/music/Jazz/Miles Davis/Kind of Blue/03 Blue in Green.flac
----

The lock file is removed after all files have been successfully written.

==== What Happens If the Service Crashes Mid-Write?

If kbeatz is killed while writing album-level tags, the `.kbeatz-write.lock` file is left on disk.
This is intentional: it signals that the album directory may contain partially-updated files.

== Write-Lock Repair

On startup, kbeatz automatically scans the library for any `.kbeatz-write.lock` files left behind by a previous crash.

=== What the Startup Repair Scan Does

For each album directory containing a `.kbeatz-write.lock` file:

. The directory is re-indexed: all FLAC files are re-read and the H2 metadata record is updated to reflect the current state of the files.
. The lock file is deleted.

The repair scan does *not* roll back any writes that completed before the crash.
Files that were already atomically renamed contain the new tag values and are kept as-is.
Files that were not yet written retain their original tag values.

=== If You Find a Lock File After a Manual Restart

If you restart kbeatz after a crash, the startup repair runs automatically. You do not need to take any manual action.
The lock file is removed as part of the repair.

If for any reason you need to identify which albums were affected, check the service logs at startup for lines matching:

[source]
----
Found N .kbeatz-write.lock file(s) - running startup repair
Repaired album directory: /path/to/album
----
