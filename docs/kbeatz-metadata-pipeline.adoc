= kbeatz Metadata Pipeline: Developer Reference
:toc: left
:toclevels: 3
:icons: font
:source-highlighter: rouge
:doctype: article

This reference covers the `.kbeatz/` hidden directory layout, the `KbeatzMetadata` JSON
schema, the multi-disc track matching algorithm, and the `localPath` image convention used
by the Discogs sync pipeline (epic #358).

See xref:adr/ADR-007-kbeatz-metadata-pipeline.adoc[ADR-007] for the design rationale.
See xref:write-lock-lifecycle.adoc[Write-Lock Lifecycle] for the `.kbeatz-write.lock` sentinel.

== Pipeline Overview

The sync pipeline runs in two decoupled phases:

----
    [Discogs API]
         |
         v
  kbeatz-sources (DiscogsMetadataSource)
    - fetch release JSON
    - map to KbeatzMetadata
    - write  .kbeatz/metadata.json
    - write  .kbeatz/source_discogs_<id>.json
         |
         v
  kbeatz-tagger (TaggerService)
    - read   .kbeatz/metadata.json
    - write  FLAC Vorbis Comments
    - embed  PICTURE blocks
----

The `.kbeatz/` directory is the handoff point between the two phases.
`kbeatz-sources` never writes FLAC files; `kbeatz-tagger` never calls network APIs.

== .kbeatz/ Directory Layout

Each album directory that has been synced at least once contains a hidden `.kbeatz/`
subdirectory:

----
<album-dir>/
  track01.flac
  track02.flac
  folder.jpg
  .kbeatz/
    metadata.json               <- provider-agnostic KbeatzMetadata (overwritten on re-sync)
    source_discogs_12345678.json <- raw Discogs API response (kept across re-syncs)
    .kbeatz-write.lock          <- write-lock sentinel (present only during an active write)
----

.File roles
[cols="1,3"]
|===
| File | Purpose

| `metadata.json`
| Provider-agnostic `KbeatzMetadata` JSON written by `kbeatz-sources` and read by
  `kbeatz-tagger`. Overwritten on every re-sync with the same or a new Discogs ID.

| `source_discogs_<discogs_id>.json`
| Raw Discogs API release response preserved for debugging and re-import.
  Filename includes the Discogs release ID so multiple IDs accumulate rather than overwrite.
  Old source files are never deleted automatically (see ADR-007 Retention Policy).

| `.kbeatz-write.lock`
| Sentinel file created before the first FLAC file in a multi-file album write and deleted
  after all files succeed. Absent under normal conditions. See
  xref:write-lock-lifecycle.adoc[Write-Lock Lifecycle].
|===

== KbeatzMetadata JSON Schema

The canonical schema is the `KbeatzMetadata` data class in
`kbeatz-common/src/main/kotlin/org/javafreedom/kbeatz/common/metadata/`.

=== Top-Level Fields

[cols="1,1,3"]
|===
| Field | Type | Description

| `schemaVersion`
| `Int`
| Schema revision. Currently `1`. Readers must reject schemas with an unsupported version.

| `album`
| `Album`
| Album-level metadata. See <<album-fields>>.

| `tracks`
| `List<Track>`

| `List<Image>`
| Images available for embedding. See <<image-fields>>.

[[localpath-convention]]
== localPath Image Convention

`Image.localPath` is a path **relative to the album directory** that the metadata mapper
(in `kbeatz-sources`) writes and `TaggerService` reads. The tagger resolves `localPath`
against the album directory to locate the file to embed.

[[localpath-table]]
.localPath conventions by picture type
[cols="1,2,2"]
|===
| pictureType (RFC 9639 s.10.7) | Description | localPath convention

| 3  | Front cover   | `folder.jpg`
| 4  | Back cover    | `back.jpg`
| 5  | Leaflet page  | `booklet-<index>.jpg` (1-based, e.g. `booklet-1.jpg`)
| 6  | Media / disc label | `disc-<index>.jpg` (1-based, e.g. `disc-1.jpg`)
| 0 (other) | Any other image | `image-<index>.jpg` (1-based, e.g. `image-1.jpg`)
|===

NOTE: `<index>` is 1-based. Indices are assigned in the order images appear in the
`images` array for that picture type.

The tagger embeds each listed image into every FLAC file in the album. It also writes
the type-3 (front cover) image as `folder.jpg` in the album directory for media player
browsing.

== Multi-Disc Subdirectory Matching

Some multi-disc albums store each disc in a subdirectory alongside the parent album
directory. For example:

----
/music/Beethoven - Symphony No. 9/
  Disc 1/
    01 - Allegro ma non troppo.flac
    02 - Molto vivace.flac
  Disc 2/
    01 - Adagio molto e cantabile.flac
    02 - Ode to Joy.flac
----

When the tagger encounters an album with `discTotal > 1` and subdirectories containing
FLAC files, it matches subdirectories to disc numbers as follows:

=== Matching Algorithm

1. Collect all immediate subdirectories that contain at least one `.flac` file.
2. Sort them **alphabetically** (case-insensitive, natural sort order).
3. Assign disc `N` to subdirectory at sort position `N-1` (0-based index).
4. Within each subdirectory, sort FLAC files alphabetically and assign track numbers
   starting from 1.

=== Error Conditions

If the number of FLAC-containing subdirectories does not match `discTotal` from
`KbeatzMetadata`, the tagger logs an `ERROR` and aborts the write for that album.
No FLAC files are modified. The failure is hard - there is no fallback.

=== Why Alphabetical Sort

Disc subdirectory names are not standardised across rippers (e.g. "CD1", "Disc 1",
"Disc One", "Part 1"). Alphabetical sort is the only ordering that works without
parsing subdirectory names. ADR-007 confirms this choice.

=== FLAC-to-Track Positional Matching

Within a disc subdirectory, FLAC files are matched to `Track` entries positionally:
the `i`-th FLAC file (0-based, alphabetical order) maps to the `i`-th track in
`metadata.json` for that disc.

If the file count does not match `trackTotal` for the disc, the tagger aborts and logs
an `ERROR`.

== Raw Discogs Backup and Re-Import

=== source_discogs_<id>.json

`kbeatz-sources` writes the raw Discogs API response to
`.kbeatz/source_discogs_<discogs_id>.json` after every successful sync. This file:

- Preserves the original Discogs release JSON for inspection and debugging.
- Allows re-deriving `metadata.json` without a network call (useful for quota management).
- Is never deleted automatically - multiple IDs accumulate as separate files.

Re-syncing the same Discogs ID overwrites the existing source file.

=== Re-Import Procedure

To re-derive `metadata.json` from a cached source file without calling Discogs:

1. Copy the desired `source_discogs_<id>.json` to a temporary location.
2. Pass it to `DiscogsMetadataSource.importFromFile(path, albumDir)` (not yet exposed as a
   CLI command - planned for a future story).

== Future Provider Convention

A MusicBrainz (or other) metadata provider must:

1. Implement `MetadataSource` in `kbeatz-sources`.
2. Produce a `KbeatzMetadata` object with `schemaVersion = 1`.
3. Write `metadata.json` to `<album_dir>/.kbeatz/` (overwriting any existing file).
4. Optionally write a `source_musicbrainz_<mbid>.json` backup file alongside.
5. Set `Album.musicbrainzId` and leave `Album.discogsId` null (or preserve it if the
   album was previously synced from Discogs).
6. Follow the `localPath` table in <<localpath-convention>> for all image entries.
7. Parse provider-specific track position strings into `discNumber` and `trackNumber`
   integers; preserve the raw string in `originalPosition`.
