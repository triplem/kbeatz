== Overview

This document records the module dependency graph, the boundary rules that govern it, and how those rules are enforced at build time. It was produced as part of Story #477 (epic #479).

|Ktor HTTP service; uses MetadataSource port and TaggerService; no kbeatz-cli dependency

|kbeatz-cli
|kbeatz-sources, kbeatz-tagger
|Fat-JAR CLI; no kbeatz-catalog dependency (see ADR-009)
|===

=== Why kbeatz-sources cannot depend on kbeatz-tagger

`kbeatz-tagger` already depends on `kbeatz-sources` (it uses `MetadataSource` to fetch release metadata). Adding a reverse dependency would create a circular dependency and is therefore prohibited.

This constraint means that any class that requires both Discogs image fetching (from kbeatz-sources) and FLAC file writing (from kbeatz-tagger) must live in a higher-level module: either kbeatz-catalog's infrastructure layer or kbeatz-cli.

== Boundary Rules

=== Rule 1: kbeatz-cli must not import from kbeatz-catalog

kbeatz-cli is a standalone CLI tool that runs without a catalog service. Importing catalog classes would couple the CLI to catalog infrastructure (H2, Exposed, Ktor), making the CLI fat and breaking the self-contained deployment model described in ADR-009.

*Enforced by*: Detekt `ForbiddenImport` rule in `kbeatz-cli/config/detekt/detekt.yml` - any import starting with `org.javafreedom.kbeatz.catalog.*` fails the build.

=== Rule 2: kbeatz-catalog application and domain layers must not import from discogs sub-packages

The catalog's domain and application layers express business concepts using the generic `MetadataSource` port (from kbeatz-sources). Importing concrete Discogs types (`DiscogsMetadataSource`, `DiscogsTokenBucket`, `DiscogsImageQuota`) into those layers would leak infrastructure concerns into business logic and make it impossible to swap in a MusicBrainz provider without touching application code.

*Allowed*: The catalog's infrastructure layer (`catalog.infrastructure.sync`) performs concrete wiring and may import discogs types - this is the intentional adapter boundary.

*Enforced by*: Detekt `ForbiddenImport` rule in `kbeatz-catalog/config/detekt/detekt.yml` - imports matching `org.javafreedom.kbeatz.sources.discogs.*` fail the build in all catalog packages except `**/infrastructure/**`.

=== Rule 3: kbeatz-sources must not import from kbeatz-tagger

This is enforced structurally by the Gradle dependency graph: kbeatz-sources does not declare kbeatz-tagger as a dependency, so any such import fails to compile.

== Album Directory Walking

The audit found three distinct walking utilities across the codebase. Each has a different purpose and technique; none is duplicated logic.

[cols="1,2,2,2"]
|===
|Location |Class / method |Technique |Purpose

|`kbeatz-catalog` +
`application/service/LibraryWalker.kt`
|`LibraryWalker.walk(libraryRoot)`
|`Files.walkFileTree()` with `SimpleFileVisitor`
|Full library scan: walks the library root, reads Vorbis Comments to build album groups for the H2 index

|`kbeatz-catalog` +
`application/service/LibraryScanService.kt`
|`collectLockFiles()`
|`Files.walkFileTree()` with `SimpleFileVisitor`
|Startup repair: finds `.kbeatz-write.lock` files left by a crashed JVM

|`kbeatz-cli` +
`util/FileUtils.kt`
|`walkDirectories(root, maxDepth)`
|`SystemFileSystem.list()` recursively
|Generic directory traversal to depth N; used by the `tag` command to find leaf album directories
|===

=== Why they are not shared

Each utility is specialised:
- `LibraryWalker` reads FLAC Vorbis Comments during the walk - it is tightly coupled to catalog indexing.
- `collectLockFiles` searches for a catalog-specific sentinel file - it belongs in catalog infrastructure.
- `walkDirectories` is a generic depth-limited traversal with no business logic - it belongs in the CLI util package.

Extracting a shared utility would force kbeatz-tagger (the only shared library that both catalog and CLI import) to include filesystem-walking logic that is only needed by the top-level consumers. The current separation is appropriate.
