== Initial Setup

=== 1. Clone and initialise

[source,bash]
----
git clone https://github.com/<owner>/kbeatz.git
cd kbeatz
chmod +x gradlew .claude/hooks/*.sh .claude/scripts/*.sh
----

=== 2. Configure environment

[source,bash]
----
cp .env.example .env        # create if not present
# Edit .env and set:
#   GITHUB_TOKEN=ghp_...     (repo + issues + packages scopes)
#   DISCOGS_TOKEN=...         (Discogs Personal Access Token)
#   LIBRARY_ROOT=/path/to/music
----

=== 3. Install MCP servers

[source,bash]
----
./.claude/scripts/setup-mcp.sh github
----

This installs `@modelcontextprotocol/server-github` and `@modelcontextprotocol/server-filesystem`
globally so Claude Code's MCP client can use them.

=== 4. Update repository references

Open `.claude/skills/create-pr/SKILL.md` and `.claude/skills/implement-epic/SKILL.md` and replace
the two `<OWNER>` / `<REPO>` placeholders with your actual GitHub organisation and repository name.

[source,bash]
----
# Quick replace: adjust owner and repo as needed
sed -i 's|<OWNER>|your-org|g; s|<REPO>|kbeatz|g' \
  .claude/skills/create-pr/SKILL.md \
  .claude/skills/implement-epic/SKILL.md
----

=== 5. Verify the build

[source,bash]
----
./gradlew buildBackends      # compiles all Kotlin services
cd kbeatz-ui && npm install && npm run build
----

=== 6. Start Claude Code

[source,bash]
----
claude
----

Claude Code will run the `SessionStart` hook which injects project context automatically.

== Starting the SDLC Workflow

Once Claude Code is running, the full AI-assisted development workflow starts with:

[source]
----
/gather-requirements
----

This launches the `RequirementsAgent` which will guide you through eliciting, challenging,
and documenting requirements before generating epics, stories, and implementation tasks.

See xref:claude-code-config.adoc[Claude Code Configuration] for the full skill reference.

== Running the Stack Locally

[source,bash]
----
podman-compose up --build

# Or with Docker Compose
docker compose up --build
----

Once running:

* Frontend: http://localhost:3005
* Backend API: http://localhost:8080/api/v1
* Swagger UI: http://localhost:8080/swagger

NOTE: kbeatz v1 runs without authentication (trusted LAN). There is no Keycloak service in v1 or v2. Keycloak JWT/OIDC is planned for Phase 3+.

== Container Deployment

This section covers running the full kbeatz stack as containers using `podman-compose`
(or `docker compose`). Both tools consume the same `compose.yml` at the project root.

=== Using Pre-built Images from GHCR

Pre-built images are published to the GitHub Container Registry on every merge to `main`.
You can pull them directly instead of building from source:

[source,bash]
----
docker pull ghcr.io/triplem/kbeatz-catalog:latest
docker pull ghcr.io/triplem/kbeatz-cli:latest
----

Pin to a specific version for reproducible deployments:

[source,bash]
----
# Use a version tag (e.g. 0.1.0)
docker pull ghcr.io/triplem/kbeatz-catalog:0.1.0

# Or pin to a commit SHA for maximum immutability
docker pull ghcr.io/triplem/kbeatz-catalog:<git-sha>
----

See the https://github.com/triplem/kbeatz/pkgs/container/kbeatz-catalog[GitHub Packages page]
for all available tags.

NOTE: GHCR does not enforce an automatic retention policy. Stale versioned tags accumulate
over time. To list and delete old images, use:
`gh api user/packages/container/kbeatz-catalog/versions --jq '.[].id'`
then `gh api -X DELETE user/packages/container/kbeatz-catalog/versions/<id>`.

=== 1. Set up environment

Copy the example env file and set the path to your FLAC library:

[source,bash]
----
cp .env.example .env
----

Edit `.env`:

[source,bash]
----
# Required: absolute path to your music library on the host
CATALOG_LIBRARY_ROOT=/home/you/Music

# Optional: get a token at https://www.discogs.com/settings/developers
DISCOGS_TOKEN=your_token_here
----

The library directory is mounted read-write at `/library` inside the catalog container.
The service starts without a `DISCOGS_TOKEN` but Discogs metadata sync will be unavailable.

=== 2. Start the stack

[source,bash]
----
podman-compose up -d       # detached

# or with Docker
docker compose up -d
----

The first run builds both images. On subsequent runs, pass `--build` only when the source
has changed.

The catalog service exposes a health check; the UI container will not start until the
catalog reports healthy (up to 90 s on first boot).

=== 3. Verify

[source,bash]
----
curl http://localhost:8080/api/v1/health
# → {"status":"UP"}
----

Then open http://localhost:3005 in a browser to reach the UI.

=== Accessing from a LAN device

The UI container runs nginx which proxies all `/api/` requests to the catalog service by
internal Docker network name. The frontend JavaScript always calls `/api/v1/...` (a relative
path), so the browser sends API requests to whichever host served the page.

To use kbeatz from a phone, tablet, or other computer on the same network:

. Find the host machine IP address (e.g. `192.168.1.10`):
+
[source,bash]
----
ip addr show | grep 'inet ' | grep -v '127.0.0.1'
----

. Open `http://<host-ip>:3005` in any browser on the LAN.

No rebuild or configuration change is needed. The nginx proxy inside the UI container
forwards API calls to the catalog service; they never leave the container network.

NOTE: The catalog API (`http://<host-ip>:8080`) does not need to be reachable from LAN
devices directly; only the UI port `3005` needs to be open on the host firewall.

=== 4. Stop the stack

[source,bash]
----
podman-compose down

# or with Docker
docker compose down
----

=== Environment variable reference

All catalog configuration lives in `kbeatz-catalog/src/main/resources/application.conf` (HOCON).
Each key can be overridden by the corresponding environment variable.

[cols="2,1,1,4"]
|===
|Environment variable |HOCON key |Required |Description

|`CATALOG_LIBRARY_ROOT`
|`catalog.libraryRoot`
|Yes
|Host path to your FLAC library. Mounted at `/library` (read-write) inside the catalog container.
The service refuses to start if this is blank.

|`CATALOG_JDBC_URL`
|`catalog.jdbcUrl`
|No
|H2 JDBC URL. Default: `jdbc:h2:file:./data/kbeatz;DB_CLOSE_DELAY=-1;MODE=PostgreSQL`.
H2 is the v1 database. PostgreSQL is the planned v2 migration target (see ADR-006).

|`CATALOG_DB_USER`
|`catalog.dbUser`
|No
|Database username. Default: `sa` (H2 default user).

|`CATALOG_DB_PASSWORD`
|`catalog.dbPassword`
|No
|Database password. Default: empty (H2 default).

|`CATALOG_DATA_DIR`
|`catalog.dataDir`
|No
|Runtime data directory for Discogs image quota tracking and other state files.
Default: `./data`.

|`DISCOGS_TOKEN`
|`catalog.discogs.token`
|No
|Discogs Personal Access Token for metadata and cover-art sync.
The service starts without it but Discogs sync is unavailable.
|===

=== Developer local overrides

Create `kbeatz-catalog/src/main/resources/local.conf` (gitignored) to override specific
keys during local development without editing `application.conf`:

[source,hocon]
----
catalog {
  libraryRoot = "/home/you/Music"
  discogs.token = "my-personal-token"
}
----

Values in `local.conf` take precedence over `application.conf`.

== CI Status Checks

All PRs must pass the GitHub Actions *CI / Build, test, and verify* status check before they can
be merged. Branch protection is configured on `main` - direct pushes are blocked.

To run the same checks locally before opening a PR:

[source,bash]
----
./gradlew check                          # backend: tests + Detekt + Kover
cd kbeatz-ui && npm run lint && npm test # frontend: ESLint + Vitest
----
